# Smart Decision 安装文档提示词集

以下提示词默认使用转换后的知识库：

- `smart-decision-7.3.0-agent-docs/AGENT_GUIDE.md`
- `smart-decision-7.3.0-agent-docs/CATALOG.md`
- `smart-decision-7.3.0-agent-docs/docs/0235-安装指南-43cc6bbd.md`

使用其他版本时，替换知识库目录，并从 `CATALOG.md` 重新定位“安装指南”，不要沿用当前版本的数字文件名前缀。

## 提示词 1：建立安装场景地图

适合首次理解一套版本的完整安装体系。

```text
你是一名产品交付架构师。请基于以下本地产品文档建立安装场景地图：

知识库：<知识库目录>
安装入口：先在 CATALOG.md 中定位“安装指南”，再沿该页面的链接阅读。

工作方法：
1. 先读取 AGENT_GUIDE.md 和 CATALOG.md，不要一次性加载全部正文。
2. 以“安装指南”的直属分支为主线，检索并补充安装概述、场景说明、
   组网规划、Hadoop 使用规划、安装流程、安装限制和安装部署附录。
3. 提取所有会改变安装路径的条件，至少检查：
   - 物理机或虚拟机
   - 标准、轻量化、工具或追加部署组网
   - 全新安装或已有系统追加部署
   - 已有 Smart Decision、Smart DataCube、Campaign、WDTN 的组合
   - 内置 FI、外置 FI、无 FI，以及是否具备 admin/OM 权限
   - EulerOS、第三方 OS，x86 或 ARM
   - 单平面、网络平面合设、IPv4/IPv6
   - 内置或外置数据库、数据库类型、是否启用客户端 SSL
   - Campaign、DataCube 设计态、大模型、营销助手、安装资产等可选能力
4. “可选”必须写明触发条件；找不到触发条件时标记为文档未说明。
5. 不要把不同场景的步骤合并成一条流程。

输出：
一、安装场景决策树
二、场景矩阵：场景、触发条件、前置系统、硬件/OS、数据平台、入口章节、限制
三、所有场景共有的安装阶段
四、各场景独有的阶段
五、必须依赖 CHM 之外资料的内容
六、文档矛盾、边界遗漏和待确认事项

证据规则：
- 每个关键结论引用 Markdown 文件路径。
- 明确标记“文档明确说明”“根据多处文档归纳”“文档未说明”。
- 遇到冲突时并列原始说法，不自行选择其中一个。
```

## 提示词 2：针对局点生成可执行安装执行手册

适合交付前根据实际环境裁剪流程。

```text
你是一名谨慎的现场交付工程师。请根据本地产品文档，为下面的局点生成安装执行手册。

知识库：<知识库目录>

局点信息：
- 产品版本：<版本>
- 安装性质：<全新安装/追加部署/迁移/重装>
- 已有产品和版本：<Smart Decision/DataCube/Campaign/WDTN/GDE 等>
- 部署载体：<物理机/虚拟机>
- 组网类型：<标准/轻量化/工具/DataCube 追加部署/待判断>
- 节点数量：<数量>
- 硬件来源：<华为/第三方>
- OS、版本、架构：<例如 EulerOS、ARM>
- 网络：<平面规划、IPv4/IPv6、是否合设、代理/离线>
- Hadoop/FI：<无/内置/外置、版本、admin/OM 权限>
- 数据库：<类型、版本、内置/外置、客户端 SSL>
- 可选能力：<Campaign/DataCube 设计态/大模型/营销助手/AI 资产等>
- 权限和其他限制：<说明>

先执行“场景完整性检查”：
1. 判断信息能否唯一确定安装分支。
2. 对会改变流程但尚未提供的信息，列出阻断问题。
3. 信息不足时只输出候选路径和待确认项，不生成伪精确步骤。

信息充分后输出：
1. 场景判定及不适用场景
2. 前置依赖和输入物清单
3. 分阶段安装执行手册：步骤、操作对象、文档依据、预期结果、检查点
4. 分支步骤及其触发条件
5. 配置参数、账号权限、端口、证书、软件包和附件
6. 安装后处理和验证
7. 失败停止条件、数据保护要求和回滚依据
8. 文档没有说明、必须由产品支持确认的事项

约束：
- 命令、路径、参数值只能来自文档；不得凭经验补写。
- 删除资源、覆盖配置、重启服务、修改数据库、证书或网络的步骤标记为高风险。
- 不要把全新安装、追加部署、物理机和虚拟机步骤混用。
- 每个步骤引用对应 Markdown 文件。
```

## 提示词 3：生成安装前检查表

适合项目开工会、环境验收和正式安装前门禁评审。

```text
请基于 <知识库目录> 为以下场景生成安装前检查表：

场景：<场景描述>

检查范围：
- 场景与组网选择
- 服务器数量、机型、CPU 架构、内存、磁盘和分区
- OS 版本、软件包、内核参数、浏览器和第三方 OS 整改
- 网络平面、网段包含关系、VIP、端口、防火墙、IPv6 和连通性
- DNS、NTP、时钟源和备份服务器
- GKit 跳板机、执行用户、权限和磁盘空间
- 场景包、软件包、APP、License、证书及校验
- FI/Hadoop 的版本、资源、用户、角色、队列、HDFS、Carbon 和客户端
- 数据库类型、版本、参数、账号、SSL 和连接验证
- 已有系统版本、依赖服务和追加部署兼容性
- 安装参数、密码限制、租户和命名空间冲突

输出 Markdown 表格：
| ID | 分类 | 检查项 | 适用条件 | 通过标准 | 检查方法 | 风险等级 | 文档来源 | 状态 |

状态固定填写“待检查”。风险等级使用：阻断、高风险、注意、建议。

要求：
1. 先锁定场景，再删除明确不适用的检查项。
2. 文档只给要求、没有给检查命令时，检查方法写“需人工确认”，不要编造命令。
3. 将所有“前提”“限制”“须知”“注意”“警告”“不支持”“切勿”纳入检查表。
4. 单独列出需要外部集成设计指导书或 Support 网站确认的项目。
5. 每项必须引用文档来源。
```

## 提示词 4：审计安装文档的冲突和遗漏

适合文档维护者或交付复盘。

```text
请审计 <知识库目录> 中的安装章节，目标是发现可能导致现场误操作的文档问题，
而不是总结安装步骤。

重点检查：
1. 概述、表格、流程图和具体步骤之间的数量或条件是否一致。
2. <、<=、>、>= 等边界是否完整，例如节点数临界值是否有归属。
3. 同名章节在物理机、虚拟机、全新安装和追加部署中是否存在差异。
4. “可选”步骤是否写清触发条件和不适用条件。
5. OS、架构、数据库、FI/Hadoop 版本和权限条件是否明确。
6. 前置步骤引用的外部文档、附件和链接是否可获得。
7. 高风险操作是否包含备份、停止条件、验证和恢复方法。
8. 安装完成标准是否覆盖管理面、数据面、可选模块和业务功能。

输出：
| 编号 | 类型 | 问题描述 | 影响场景 | 证据 A | 证据 B | 风险 | 建议澄清方式 |

类型包括：矛盾、边界缺失、条件缺失、步骤缺失、外部依赖、失效引用、风险说明不足。

规则：
- 只有存在文档证据时才报告问题。
- 不把个人偏好当成缺陷。
- 对无法确认的问题使用“疑似”，并写明还需要什么证据。
```

## 提示词 5：根据安装故障定位原因

适合现场安装失败后的只读分析，不授权 Agent 执行修复。

```text
请根据 <知识库目录> 分析以下安装故障：

场景信息：<部署方式、组网、OS/架构、FI、数据库、安装阶段>
故障现象：<现象>
错误信息：<原始报错，不要改写>
已执行操作：<操作>
日志位置或日志片段：<内容>

分析流程：
1. 确认故障属于哪个安装阶段和场景分支。
2. 搜索安装步骤、安装限制、安装日志、FAQ 和相关组件故障章节。
3. 将候选原因按“证据匹配度”和“检查成本”排序。
4. 先给只读检查方法，再给文档明确提供的修复建议。
5. 涉及删除 Pod、重装、覆盖配置、数据库变更、证书替换或服务重启时，
   不执行操作，只标记风险并要求人工确认。

输出：
- 场景和阶段判断
- 最可能原因及证据
- 依次执行的只读检查
- 每项检查结果如何影响下一步判断
- 文档明确给出的修复方案
- 高风险动作及前置备份要求
- 仍缺少的信息

每个原因和修复建议必须引用 Markdown 文件。无法从文档证明时写“推测”。
```
