# 按场景规划产品安装

使用确定性的安装模型选择主路线。使用 Agent 消除歧义、读取已选路线的证据并解释细节。

## 定位版本化资产

1. 定位包含 `AGENT_GUIDE.md`、`CATALOG.md`、`manifest.json` 和 `docs/` 的转换后知识库。
2. 定位 `installation-models/<产品>/<版本>/` 下对应的模型。
3. 确认模型版本与用户要求的产品版本一致。读取 `model.lock.json`；如果 CHM 哈希或版本不一致，明确提示。
4. 如果存在多个版本且用户没有指定版本，询问应使用哪个版本。

## 查看支持的组合

运行：

```text
python -m chm_agent scenarios <模型目录>
python -m chm_agent validate-model <模型目录> --knowledge-base <知识库目录>
```

模型校验失败时停止。回答安装问题时，不得在未告知用户的情况下修改模型。

## 建立场景档案

阅读[场景档案说明](references/scenario-profile.md)。将用户的自然语言归一化为 `scenarios` 输出的准确字段 ID 和取值。

优先确定会选择不同主路线的字段：

- `installation_nature`；
- `deployment_carrier`；
- 追加部署时的 `base_environment`。

然后记录组网、硬件、操作系统、CPU 架构、数据平台及其访问方式、数据库及 SSL 模式、网络和可选模块。

只询问编译器报告为阻断项的缺失信息。其他未知项保留为条件分支。

## 先编译，再检索细节

使用 JSON 场景档案或明确的字段值运行编译器：

```text
python -m chm_agent plan <模型目录> --profile <场景档案.json>
```

或者：

```text
python -m chm_agent plan <模型目录> --set installation_nature=new --set deployment_carrier=physical
```

按以下方式解释结果：

- `ready`（就绪）：使用已选路线及适用步骤。
- `needs_input`（需要补充信息）：展示阻断项；如果已选出稳定的主路线，则保留该路线，并将未决步骤标记为条件步骤。
- `invalid`（组合无效）：结合来源解释违反的约束并停止，不得生成可执行的安装执行手册。

不得仅因关键词检索命中某个步骤，就将编译路线中不存在的步骤加入方案。

## 补充已编译步骤的细节

对每个已选步骤或条件步骤执行以下操作：

1. 打开编译器列出的全部来源 Markdown 文件。
2. 只沿已选场景分支或明确共用页面中的链接继续阅读。
3. 提取前置条件、顺序操作、命令、参数、警告、预期结果和验证细节。
4. 在安装限制和附录中检索同一组件或操作。
5. 将无法访问的外部手册、电子表格、Support 页面或附件标记为外部依赖。

只使用 `rg` 补充已编译步骤的细节，不得用它判断路线适用性。

## 生成最终安装执行手册

按以下结构输出：

1. 场景档案以及模型、文档版本。
2. 适用性判定及被排除的相邻路线。
3. 阻断性前置条件。
4. 有序的端到端安装步骤。
5. 带触发条件和汇合点的条件分支。
6. 安装前检查表。
7. 安装后验证。
8. 风险、停止条件、回滚依据和文档冲突。
9. 待确认问题和外部依赖。

步骤使用以下表格：

| 编号 | 阶段 | 操作与细节 | 适用条件 | 预期结果 | 风险 | 来源 |
|---|---|---|---|---|---|---|

风险等级使用 `阻断`、`高风险`、`注意` 和 `建议`。

## 证据与安全要求

- 每项重要要求、命令、参数、限制和验证步骤都必须引用本地 Markdown 路径。
- 明确区分 `文档明确说明`、`模型约束`、`跨章节归纳` 和 `文档未说明`。
- 将模型文件视为已评审的路线选择知识，将 Markdown 文件视为操作步骤证据。
- 不得编造兼容性、命令、参数值、回滚过程或验收标准。
- 除非用户另行明确要求，否则不得执行安装操作。
- 在删除、重装、删除 Pod、修改数据库、替换证书、修改网络或重启服务之前，必须列出文档规定的备份要求和停止条件。

## 未建模版本的处理方式

如果没有匹配的模型，不得声称已完整覆盖所有组合。根据安装指南的章节层次生成临时路线，将其标记为 `未建模`，列出全部未决的路线选择条件，并建议先创建和评审版本化模型，再用于现场执行。
